# Подробный отчёт об обучении модели

Документ описывает полный ML-пайплайн проекта: данные, preprocessing, feature engineering, обучение моделей, результаты, анализ ошибок, риски и артефакты запуска.

## 1. Постановка задачи

Проект решает задачу антифрода для банка или платежного сервиса. Цель модели — присваивать каждой онлайн-транзакции fraud score, чтобы:

- снижать прямые потери от мошенничества;
- уменьшать число ненужных ручных проверок;
- повысить долю подозрительных операций, которые выявляются до подтверждения.

В качестве реалистичной бизнес-гипотезы для проекта можно рассматривать сценарий, в котором модель позволяет снизить fraud loss на `10-15%`, сократить объем ручной проверки на `15-20%` и повысить долю действительно подозрительных операций в верхней части очереди на проверку.

## 2. Данные

Используется публичный датасет **IEEE-CIS Fraud Detection**.

### 2.1. Источники

- `train_transaction.csv`
- `train_identity.csv`

### 2.2. Объединение

Датасет состоит из двух связанных таблиц:

- `train_transaction.csv` — основная таблица транзакций;
- `train_identity.csv` — дополнительная таблица identity- и device-сигналов.

Обе таблицы объединяются по ключу `TransactionID` с помощью `left join`, где за базовую таблицу берётся `train_transaction.csv`.

Это означает:

- каждая строка в итоговом датасете соответствует одной транзакции;
- все транзакции из `train_transaction.csv` сохраняются;
- identity-признаки подтягиваются только там, где для соответствующего `TransactionID` есть данные в `train_identity.csv`;
- если identity-информации нет, соответствующие колонки остаются пустыми и далее обрабатываются в preprocessing.

Такой способ объединения логичен для антифрод-задачи: именно транзакция является объектом классификации, а identity-таблица лишь добавляет к ней контекст.

### 2.3. Какие данные используются

- сумма транзакции;
- время транзакции;
- карточные признаки `card*`;
- адресные признаки `addr*`, `dist*`;
- email-домены;
- поведенческие признаки `C*`, `D*`, `M*`;
- identity- и device-признаки;
- часть `V*`-признаков с наименьшей долей пропусков.

### 2.4. Таргетная колонка

Целевая переменная в задаче — `isFraud`.

Интерпретация:

- `isFraud = 1` — транзакция является мошеннической;
- `isFraud = 0` — транзакция считается легитимной.

Именно этот столбец используется при стратифицированном разбиении данных, обучении моделей и расчёте всех quality-метрик.

### 2.5. Почему исходно колонок 400+, а модель использует 127 признаков

После объединения transaction и identity таблиц исходное пространство признаков очень широкое: в merged-датасете оказывается несколько сотен колонок. Это связано с тремя факторами:

1. В `train_transaction.csv` уже много табличных признаков, включая группы `C*`, `D*`, `M*`, `V*`.
2. `train_identity.csv` добавляет отдельный набор identity- и device-колонок.
3. После merge все эти столбцы попадают в единое пространство, даже если часть из них сильно разрежена или содержит слишком много пропусков.

Однако в модель подаются не все исходные колонки, а только отобранный поднабор признаков.

Причины сокращения размерности до итоговых 127 признаков:

- изначально берётся только фиксированный список содержательно полезных групп: `TransactionAmt`, `TransactionDT`, `ProductCD`, `card*`, `addr*`, `dist*`, `C*`, `D*`, `M*`, `id_*`, `DeviceType`, `DeviceInfo`;
- из всех `V*`-признаков выбираются только `max_v_features = 20` колонок с наибольшей долей непустых значений;
- часть исходных полей не используется, если она не входит в заранее заданные группы;
- затем поверх выбранных полей строятся новые engineered features, и именно финальный набор после preprocessing и feature engineering образует матрицу признаков для модели.

Итоговые 127 признаков — это не “127 сырых колонок из CSV”, а результат отбора полезных исходных полей плюс автоматически построенные дополнительные признаки.

## 3. Препроцессинг

В пайплайне реализованы следующие шаги:

1. Проверка наличия обязательных CSV в `data/raw/`.
2. Merge transaction и identity таблиц.
3. Обработка пропусков.
4. Кодирование категориальных значений.
5. Подготовка train и validation выборок.

### 3.1. Заполнение пропусков

Пропуски обрабатываются отдельно для числовых и категориальных признаков.

- Для категориальных признаков все пропущенные значения заменяются на строковый токен `missing`.
- Для числовых признаков после построения полного набора признаков используется заполнение медианой, посчитанной только по train-части.
- Если при расчете медианы значение не определено, используется `0.0`.

Такой подход делает обработку стабильной и одинаковой для train, validation и будущего inference.

### 3.2. Кодирование категориальных переменных

Категориальные признаки кодируются через словари соответствия `category -> integer index`, которые строятся на train-части после заполнения пропусков значением `missing`.

Алгоритм кодирования:

1. Для каждого категориального признака берутся все значения train-выборки.
2. Значения переводятся в строковый вид.
3. Считается частота встречаемости категорий.
4. Категории сортируются по частоте.
5. Каждой категории присваивается индекс `0, 1, 2, ...`.
6. Неизвестные категории на validation или inference получают значение `-1`.

Это не one-hot encoding: используется компактное целочисленное кодирование, которое подходит для дерева решений и снижает размерность пространства признаков.

### 3.3. Разбиение на train и validation

Разбиение выполняется функцией `train_test_split` со следующими параметрами:

- `test_size = 0.2`
- `random_state = 42`
- `stratify = isFraud`

То есть 80% данных идут в обучение и 20% в validation, а доля мошеннических транзакций сохраняется в обеих частях выборки.

## 4. Feature engineering

### 4.1. Базовые признаки

- `TransactionAmtLog`
- `TransactionHour`
- `TransactionDay`
- `TransactionWeek`
- `EmailDomainMatch`
- `MissingValueCount`
- `HasDeviceInfo`
- `HasDeviceType`

### 4.2. Frequency encoding

Частотные признаки строятся для:

- `card1`, `card2`, `card3`, `card5`
- `addr1`, `addr2`
- `P_emaildomain`, `R_emaildomain`
- `DeviceInfo`

Алгоритм построения frequency encoding:

1. Для каждой выбранной сущности берётся train-колонка.
2. Все пропуски заменяются на `missing`.
3. Для каждого уникального значения считается число его появлений в train.
4. Это число подставляется обратно в каждую строку как новый числовой признак вида `feature_freq`.

Например, если конкретный `card1` встретился в train `12 500` раз, то соответствующий признак `card1_freq` для всех строк с этим `card1` будет равен `12500`.

### 4.3. Агрегатные признаки

Для ключевых сущностей считаются:

- количество транзакций;
- средняя сумма транзакции.

Это делается для:

- `card1`
- `addr1`
- `DeviceInfo`
- `P_emaildomain`

Алгоритм построения агрегатных признаков:

1. Для выбранной сущности формируется группировка по train-выборке.
2. Внутри каждой группы по `TransactionAmt` считаются:
   - количество строк в группе;
   - среднее значение суммы транзакции.
3. Затем эти агрегаты маппятся обратно в каждую строку.

В результате появляются признаки вида:

- `card1_tx_count`
- `card1_amt_mean`
- `addr1_tx_count`
- `addr1_amt_mean`

Такие признаки позволяют модели учитывать не только конкретное значение сущности, но и его поведенческий контекст в обучающей выборке.

## 5. Модели

### 5.1. Baseline

В качестве baseline используется `LogisticRegression`.

Задача baseline — показать, насколько сильно можно улучшить качество за счёт более гибкой модели и feature engineering.

Конфигурация baseline:

- `StandardScaler`
- `LogisticRegression`
- `max_iter = 500`
- `class_weight = balanced`
- `solver = lbfgs`

### 5.2. Основная модель

Основной моделью выбрана `LightGBM`, потому что:

- она хорошо работает на табличных данных;
- устойчива к сложным нелинейным зависимостям;
- даёт сильное качество при разумном времени обучения;
- подходит для антифрод-задач с большим числом признаков.

### 5.3. Схема обучения и подбор параметров

В текущей реализации не используется отдельный автоматический search по гиперпараметрам вроде `GridSearchCV` или `Optuna`. Вместо этого применяется ручной фиксированный набор параметров, подобранный как разумный старт для табличной антифрод-задачи.

Используемая конфигурация `LightGBM`:

- `n_estimators = 400`
- `learning_rate = 0.05`
- `num_leaves = 64`
- `max_depth = -1`
- `subsample = 0.8`
- `colsample_bytree = 0.8`
- `reg_alpha = 0.1`
- `reg_lambda = 0.1`
- `random_state = 42`
- `n_jobs = -1`

Отдельно учитывается дисбаланс классов. Для этого вычисляется

`scale_pos_weight = (1 - positive_rate) / positive_rate`

на основе доли положительного класса в train-выборке.

После обучения для обеих моделей строится распределение вероятностей на validation-части. Затем рабочий порог выбирается автоматически по максимуму `F1` на validation.

## 7. Анализ ошибок

### 7.1. False positives

Типичный false positive означает, что модель посчитала честную транзакцию подозрительной. Это может происходить, если:

- транзакция проходит с редкого устройства;
- поведение клиента нетипично;
- есть редкое сочетание признаков, похожее на фрод.

Пример из валидации:

- `TransactionAmt = 9.468`
- `ProductCD = C`
- `DeviceType = mobile`
- `P_emaildomain = gmail.com`
- `R_emaildomain = gmail.com`
- `fraud_probability = 0.9881`

Гипотеза: модель увидела сочетание признаков, характерное для подозрительных C-транзакций, и переоценила риск несмотря на то, что операция была легитимной. Это типичный пример ложного срабатывания, которое потенциально увеличивает нагрузку на ручную проверку.

### 7.2. False negatives

Типичный false negative означает, что мошенническая транзакция была пропущена. Это может происходить, если:

- поведение злоумышленника похоже на нормальное;
- не хватает identity-сигналов;
- новый паттерн мошенничества ещё не отражён в обучении.

Пример из валидации:

- `TransactionAmt = 50.0`
- `ProductCD = H`
- `DeviceType = desktop`
- `P_emaildomain = gmail.com`
- `R_emaildomain = aol.com`
- `fraud_probability = 0.1523`

Гипотеза: транзакция выглядела слишком близко к нормальному пользовательскому поведению и не содержала достаточно сильных identity-сигналов, поэтому модель недооценила риск. Это опасный тип ошибки, потому что он ведёт к прямым финансовым потерям.

## 9. Риски и мониторинг

### 9.1. Риски

- `Data drift`. Распределения входных признаков могут со временем меняться: появляются новые типы устройств, смещаются суммы транзакций, изменяется структура email-доменов и адресных признаков. Это постепенно ухудшает качество модели, даже если сама логика мошенничества не изменилась.
- `Concept drift`. Поведение злоумышленников адаптируется к антифрод-правилам и модели. В результате паттерны, на которых модель обучалась, перестают быть актуальными, а старая decision boundary начинает пропускать новые сценарии фрода.
- Рост ложных блокировок. Даже качественная модель может начать отправлять в review или block слишком много честных клиентов, если изменится профиль пользователей или бизнес-процесс пороговой обработки score.
- Неполные identity-данные. Часть признаков из `train_identity.csv` часто пропущена. В production это может быть ещё выраженнее: не все устройства, браузеры и дополнительные сигналы доступны стабильно.
- Регуляторные и explainability-риски. Если решение модели влияет на клиентский путь, банк или платежный сервис должен понимать, как объяснять алерты и как обосновывать автоматические ограничения.
- Риск переобучения на публичный датасет. Публичная структура данных удобна для обучения, но не гарантирует, что тот же профиль зависимостей сохранится в реальном промышленном потоке транзакций.

### 9.2. Что мониторить в проде

- Долю алертов: сколько транзакций попадает в review или block.
- Долю подтвержденного фрода: насколько хорошо верхняя часть risk queue действительно содержит мошеннические операции.
- Precision ручной очереди: насколько полезна модель для операционной antifraud-команды.
- Распределения ключевых признаков: `TransactionAmt`, `card1_freq`, `addr1_freq`, `DeviceInfo`, `P_emaildomain` и другие важные сигналы.
- Стабильность fraud score: нет ли смещения всего распределения вероятностей по сравнению с историческим уровнем.
- Частоту ошибок по сегментам: например, по `ProductCD`, типу устройства, часовым окнам, группам сумм.
